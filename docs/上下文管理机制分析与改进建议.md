# 上下文管理机制分析与改进建议

> 本文档基于与 AI 编程助手的对话总结，分析了当前项目（DS Agent）的提示词/上下文管理机制、其局限性，并提出改进方向。

## 一、当前机制概述

### 1.1 网页会话控制的两个方法

在 `src/renderer/api/DeepSeekClient.js` 中，定义了两种与 DeepSeek 网页交互的方式：

- **`sendRaw(prompt)`**：调用 `_clickNewChatButton()` 强制开启一个新网页会话，然后注入提示词。
- **`sendContinuation(prompt)`**：不点击“新会话”，直接在现有网页会话中输入提示词。

上层通过标志 `isContinuation` 选择调用哪个方法。

### 1.2 pi 的决策逻辑

在 `src/main/llm-bridge.js` 的 `_dispatch` 方法中，pi 维护了一个内部会话状态 `_session`：

```javascript
this._session = { messageCount: 0, totalChars: 0, active: false };
```

判断是否续接（`isContinuation = true`）的条件：

```javascript
if (this._session.active && this._session.totalChars < MAX_SESSION_CHARS) {
  if (messages.length > this._session.messageCount) {
    isContinuation = true;
  }
}
```

其中 `MAX_SESSION_CHARS = 1500000`（约 500k tokens，按 3 字符/token 估算）。

- **首次请求**：`active=false` → 新建会话。
- **上次请求成功 + 未超字符限制 + 消息数增加** → 续接（复用网页会话）。
- **上次请求成功 + 未超限 + 消息数未增加**（如重发相同历史） → 新建会话。
- **上次请求成功 + 已超限** → 新建会话。
- **上次请求失败（含 stall 卡死）** → 新建会话。

成功完成后更新会话状态：

```javascript
this._session.messageCount = messages.length;
this._session.totalChars += state.handle.prompt.length;
this._session.active = true;
```

失败时（如 stall）设置 `active = false`，下次请求强制新建。

### 1.3 prompt 构建方式

- **新建会话（`isContinuation = false`）**：调用 `buildPrompt(body, { template })`，将 **完整的** `messages` 数组（即整个对话历史）序列化为 prompt 文本。
- **续接（`isContinuation = true`）**：调用 `buildContinuationPrompt(messages, startIndex)`，只格式化增量消息，避免重复注入约束/工具/协议。

## 二、核心问题：新建会话并不减少 token 消耗

### 2.1 根本矛盾

- **`MAX_SESSION_CHARS` 仅用于决定是否“新建网页会话”**，并不对历史消息做任何压缩或截断。
- 无论是新建还是续接，发送给 DeepSeek 的 prompt 都包含 **完整的对话历史**（新建时是完整历史，续接时也是完整历史，只是构建方式略有差异）。
- DeepSeek 网页版的后端模型具有 1M token 的上下文窗口。当历史 token 数接近或超过 1M 时，即便新建一个网页会话，模型仍然会收到超长的 prompt，导致：
  - 早期消息被截断（模型实际能看到的上下文变短）。
  - 或请求直接失败（取决于后端实现）。

因此，**“新建网页会话”只是 UI 层面的会话重置，并不能从根本上解决 token 超限问题**。

### 2.2 用户体感问题

- 当历史过长时，用户可能发现 AI 忘记了早期的对话内容（因为被截断）。
- 网页界面上也看不到之前的聊天记录（因为是新建窗口），体验不佳。

## 三、改进方向

### 3.1 自动压缩（推荐）

在发送请求前，检测当前历史的总 token 数（使用精确的 tokenizer，如 `gpt-tokenizer` 或 `tiktoken` 针对 `deepseek-chat` 模型）。

当总 token 超过阈值（例如 800k，预留 200k 给模型回复）时，自动触发压缩：

1. 调用 LLM（可以是 DeepSeek 自身）对历史进行摘要。
2. 将摘要作为系统消息的一部分，并保留最近的 N 轮完整对话（例如最后 10 轮）。
3. 用压缩后的消息数组替换原来的 `messages`。
4. 新建网页会话（`isContinuation=false`）发送压缩后的历史。

压缩本身也会消耗 token，因此需要控制摘要的简洁性，或提示用户压缩中。

### 3.2 用户手动触发压缩

提供一个新的 slash 命令，例如 `/compact`，由用户主动压缩当前对话历史。压缩后的对话成为新会话的开端，网页 UI 上可回填压缩后的摘要（可选）。

### 3.3 滑动窗口（简单但会丢失信息）

保留最近 K 轮完整消息，丢弃更早的消息。适用于临时性任务，不适用于需要长期记忆的场景。

### 3.4 警告 + 手动决策

当历史 token 数超过阈值时，通过终端提示或弹窗警告用户：

> “当前对话已接近上下文窗口上限（约 1M tokens），继续发送可能导致早期消息被截断。建议执行 `/compact` 压缩历史，或 `/clear` 清空历史后重试。”

用户确认后（例如输入 `continue`）才发送。

### 3.5 精确 token 计数

无论采用哪种方案，首先需要精确统计 token 数。推荐使用：

- `tiktoken`（OpenAI 官方库，支持多种编码，可配置为 `cl100k_base` 近似 DeepSeek 的编码）。
- 或 `@dqbd/tiktoken` 的轻量版本。

在 `LlmBridge` 中增加一个方法 `estimateTokens(text)` 使用真实编码器。

## 四、实施建议

### 4.1 短期（最小改动）

- 增加精确 token 计数，在请求前检查历史 token 数。
- 若超过阈值（如 800k），打印警告日志，并在终端输出提示，要求用户执行 `/compact` 或 `/clear`。
- 用户可手动输入命令来处理。

### 4.2 中期（半自动化）

- 实现 `/compact` 命令，调用 LLM 压缩历史。
- 在压缩成功后将新消息数组保存到会话中，并重置 `_session`（强制新建网页会话）。
- 如果用户未压缩且继续发送超限请求，再次警告并等待确认。

### 4.3 长期（全自动）

- 当 token 超限时自动触发压缩，无需用户干预。
- 压缩过程在后台进行，完成后自动发送请求。
- 在终端显示“历史已压缩，继续对话…”。

### 4.4 注意事项

- 压缩操作本身会消耗 token 和时间，可能导致响应延迟。
- 需确保压缩摘要不丢失关键信息（可让用户预览压缩结果）。
- 压缩后，原有的工具调用结果、系统指令等需要保留完整性。

## 五、总结

当前项目的上下文管理机制能够自动选择是否新建网页会话，但其 token 限制判断仅用于 UI 层面，无法解决模型上下文窗口溢出的根本问题。通过引入精确 token 计数与自动/手动压缩历史，可以真正保证长对话的连续性和完整性，提升用户体验。
