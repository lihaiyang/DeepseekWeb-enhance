# 停止检测方案

> 本文档分析 DeepSeek 网页回复中断导致 pi 卡死的根因，提出基于 DOM 信号的三层检测方案，并规划如何对标 OpenAI 原生 API 的错误协议。
>
> 创建日期：2026-05-29

## 一、问题描述

### 1.1 现象

当 DeepSeek 网页端回复中途断开（后端超时、内容安全截停、网络中断等），pi 会进入无限等待状态，表面现象为"卡住不动"。

### 1.2 根因：错误被静默吞掉

错误传播链路中存在三个"静默断层"：

#### 断层 1：SSE 流结束被无条件当作成功

网络钩子（`src/preload/index.js`）拦截了 DeepSeek 的 fetch 和 XHR 请求。当 SSE 流关闭时（`result.done` 或 `readyState === 4`），**无条件调用 `onEnd`**，整条链路认为回复成功完成：

```javascript
// fetch hook — pump()
if (result.done) {
  // ...
  lastLens = window.__dsAgentFireCallbacks(
    thinkingAcc, responseAcc, true,  // ← isFinal = true，无条件
    ...
  );
  return;
}
```

不管 DeepSeek 后端是真的正常结束，还是中途被截断，对上层来说都是 `finish_reason: 'stop'`。

#### 断层 2：stall watchdog 在首 chunk 到达前不生效

`LlmBridge` 中的 stall watchdog（5s 超时）只在收到**首条** `thinking` 或 `content` 事件后才启动：

```javascript
// llm-bridge.js — _onIncoming
if (kind === 'thinking') {
  translator.pushReasoning(payload.delta);
  this._armStallTimer(state);   // ← 首条 thinking 之后才开始计时
}
```

如果 DeepSeek 在发出任何内容之前就中断了（页面崩溃、网络断开），没有任何计时器在跑，pi 在 HTTP 流上无限等待。

#### 断层 3：错误被伪装成正常内容

即使 stall watchdog 触发，错误也是作为 **content delta** 注入到流中，而非 SSE error 事件：

```javascript
// parse-stream.js — translator.fail
state.markError('stalled');
translator.fail(STALL_MESSAGE);  // ← 把错误消息注入 content
translator.end();                 // ← 正常结束流
```

pi 看到的是一个"正常完成"的回复，只不过内容里包含了一句错误话。pi 无法区分"真实回复"和"错误伪装"，可能继续用不完整的回复做后续决策（解析工具调用、信任分析结论等）。

### 1.3 当前行为 vs OpenAI 标准协议

| 场景 | OpenAI 标准行为 | 当前行为 | pi 感知 |
|------|----------------|---------|---------|
| 正常完成 | `data: {...}` ... `data: [DONE]` | 一致 ✅ | 正常工作 |
| 桥内部错误 | `data: {"error":{...}}` 然后关闭 | 直接 `res.end()`，不写 error 也不写 [DONE] | `ECONNRESET` 或 `IncompleteRead` |
| stall 5s 超时 | N/A | `translator.fail()` → 错误注入 content | 收到含错误消息的正常回复 |
| 首 chunk 永远不到 | 服务端超时 → 关闭连接 | **无限等待** | pi 卡死 |
| DeepSeek 中途断流 | SSE error 或连接关闭 | **onEnd 正常触发** | 当作成功 |

## 二、核心思路：用 DeepSeek 自己的信号替代猜测

### 2.1 关键发现

DeepSeek 网页在回复中断时，会在 DOM 中渲染 **"已停止"** 文案。这是一个**确定性信号**——DeepSeek 自己告诉我们这次回复没完成。

### 2.2 为什么比现有机制好

| 机制 | 原理 | 延迟 | 覆盖率 | 可靠性 |
|------|------|------|--------|--------|
| stall watchdog | 猜测（5s 无新数据） | 5s | 部分（需首 chunk 先到） | 低（误判/漏判） |
| 首 chunk 超时 | 猜测（30s 无首 chunk） | 30s | 覆盖首 chunk 缺失场景 | 中（阈值难调） |
| **DOM "已停止" 检测** | DeepSeek 确定性信号 | < 100ms | 覆盖绝大部分中断场景 | **高** |

### 2.3 三层防护设计

```
第一层：DOM "已停止" 检测（最快、最准）
  ↓ 未触发（网络断开等极端情况）
第二层：stall watchdog（5s chunk 间超时）
  ↓ 未触发（首 chunk 不到）
第三层：首 chunk 超时（30s）
```

## 三、具体实现

### 3.1 dom-bridge.js：增加中断检测函数

在 `src/renderer/deepseek/dom-bridge.js` 中增加：

```javascript
/**
 * 检测 DeepSeek 页面是否显示回复被中断的信号。
 * DeepSeek 网页在回复中断时会在 DOM 中渲染"已停止"等文案。
 * @returns {boolean}
 */
checkResponseInterrupted: function () {
  var body = document.body;
  if (!body) return false;

  // 方法 1：全文搜索中文"已停止"
  // 覆盖："已停止"、"已停止生成" 等变体
  var text = body.textContent || '';
  if (/已停止/.test(text)) return true;

  // 方法 2（备选）：英文界面
  if (/stopped/i.test(text)) return true;

  // 方法 3（可选）：检查特定状态元素
  // 如果 DeepSeek 页面有固定 class/data 属性标记停止状态，
  // 可以在这里做更精确的匹配：
  // var statusEl = document.querySelector('[data-status="stopped"]');
  // if (statusEl) return true;

  return false;
}
```

### 3.2 sse-parser.js：增加 onError 回调

在 `src/renderer/deepseek/sse-parser.js` 中，`__dsAgentSSECallbacks` 增加 `onError` 字段：

```javascript
window.__dsAgentSSECallbacks = {
  onThinking: null,
  onContent: null,
  onEnd: null,
  onError: null,  // 新增：流异常中断时的回调
};
```

### 3.3 adapter.js：处理 onError 回调

在 `src/renderer/deepseek/adapter.js` 的构造函数中注册 `onError`：

```javascript
window.__dsAgentSSECallbacks.onError = function (message) {
  if (self._pendingReject) {
    var reject = self._pendingReject;
    self._clearPending();
    reject(new Error(message || 'DeepSeek 回复被中断'));
  }
};
```

### 3.4 preload/index.js：修改网络钩子

**修改 fetch hook 的 `pump()` 函数：**

```javascript
// 原代码：
if (result.done) {
  if (buffer.trim()) {
    var lines = buffer.split("\n");
    for (var i = 0; i < lines.length; i++)
      window.__dsAgentProcessLine(lines[i], thinkingAcc, responseAcc, pTracker);
  }
  lastLens = window.__dsAgentFireCallbacks(
    thinkingAcc, responseAcc, true,
    lastLens.thinkingLen, lastLens.responseLen, pTracker
  );
  return;
}

// 改为：
if (result.done) {
  if (buffer.trim()) {
    var lines = buffer.split("\n");
    for (var i = 0; i < lines.length; i++)
      window.__dsAgentProcessLine(lines[i], thinkingAcc, responseAcc, pTracker);
  }
  // 先发射已累积的内容 delta
  lastLens = window.__dsAgentFireCallbacks(
    thinkingAcc, responseAcc, false,  // ← isFinal = false，先不结束
    lastLens.thinkingLen, lastLens.responseLen, pTracker
  );
  // 等一个微任务让 DOM 更新，然后检测中断信号
  setTimeout(function () {
    var dom = window.__dsAgentDOM;
    var interrupted = dom && dom.checkResponseInterrupted &&
                      dom.checkResponseInterrupted();
    if (interrupted) {
      console.error(PREFIX + " DeepSeek 回复被中断（检测到"已停止"信号）");
      var cb = window.__dsAgentSSECallbacks;
      if (cb && typeof cb.onError === 'function') {
        cb.onError('DeepSeek 回复被中断');
      }
    } else {
      // 正常完成
      lastLens = window.__dsAgentFireCallbacks(
        thinkingAcc, responseAcc, true,
        lastLens.thinkingLen, lastLens.responseLen, pTracker
      );
    }
  }, 100);
  return;
}
```

**修改 XHR hook 的 `readystatechange`：**

```javascript
// 在 this.readyState === 4 的分支中，同样增加"已停止"检测
if (this.readyState === 4) {
  if (xhrBuffer.trim()) {
    var lines = xhrBuffer.split("\n");
    for (var k = 0; k < lines.length; k++)
      window.__dsAgentProcessLine(lines[k], thinkingAcc, responseAcc, xhrPTracker);
    xhrBuffer = "";
  }
  // 发射已累积内容
  lastLens = window.__dsAgentFireCallbacks(
    thinkingAcc, responseAcc, false,
    lastLens.thinkingLen, lastLens.responseLen, xhrPTracker
  );
  // DOM 检测
  var self = this;
  setTimeout(function () {
    var dom = window.__dsAgentDOM;
    var interrupted = dom && dom.checkResponseInterrupted &&
                      dom.checkResponseInterrupted();
    if (interrupted) {
      console.error(PREFIX + " XHR: DeepSeek 回复被中断");
      var cb = window.__dsAgentSSECallbacks;
      if (cb && typeof cb.onError === 'function') {
        cb.onError('DeepSeek 回复被中断');
      }
    } else {
      lastLens = window.__dsAgentFireCallbacks(
        thinkingAcc, responseAcc, true,
        lastLens.thinkingLen, lastLens.responseLen, xhrPTracker
      );
    }
  }, 100);
}
```

### 3.5 deepseek-bridge.js：增加 onError 的 IPC 转发

在 `src/renderer/deepseek-bridge.js` 中注册 adapter 的 error 回调（当前只有 thinking/content 的注册）。DeepSeekClient 接管 adapter 回调时需要保留 error 回调的转发。

### 3.6 http-server.js：SSE error 事件写入

当 `bridge.request` 抛出异常时，写入 OpenAI 兼容的 SSE error 事件：

```javascript
// 当前代码：
} catch (err) {
  log('llm:error', { message: err.message });
  // Don't write [DONE] — let the SSE connection close abruptly
}

// 改为：
} catch (err) {
  log('llm:error', { message: err.message });
  // 写入 OpenAI 兼容的 SSE error 事件
  var errorChunk = {
    error: {
      message: err.message || 'unknown error',
      type: 'server_error',
    },
  };
  try { res.write('data: ' + JSON.stringify(errorChunk) + '\n\n'); } catch (_) {}
}
```

### 3.7 llm-bridge.js：首 chunk 超时

在 `_launch` 中增加首 chunk 到达前的超时保护（30s）：

```javascript
var FIRST_CHUNK_TIMEOUT_MS = 30000;
var firstChunkReceived = false;

var firstChunkTimer = setTimeout(function () {
  if (!firstChunkReceived && !state.finalized) {
    _onStall(state, '首 token 超时（30s 无响应）');
  }
}, FIRST_CHUNK_TIMEOUT_MS);

// 在 _onIncoming 收到首条 thinking/content 时清除
if (!firstChunkReceived) {
  firstChunkReceived = true;
  clearTimeout(firstChunkTimer);
}
```

### 3.8 parse-stream.js：stall 改为 reject 而非内容注入

```javascript
// 当前：translator.fail 把错误注入 content delta
// 改为：直接调用 error 回调，让整条链路走错误路径
translator.fail = function (message) {
  state.markError(message);
  // 不再注入 content，改为通过 onError 回调传播
  // （需要在 createTranslator 中增加 onError 参数）
};
```

## 四、实施优先级

| 优先级 | 改动 | 效果 |
|-------|------|------|
| 🔴 P0 | preload 网络钩子 → DOM "已停止" 检测 | 解决最主要的中断卡死问题 |
| 🔴 P0 | sse-parser + adapter + bridge → onError 链路 | 让错误信号正确传播到 main 进程 |
| 🔴 P0 | http-server → 异常时写 SSE error 事件 | 对齐 OpenAI 错误协议，让 pi 感知失败 |
| 🟡 P1 | llm-bridge → 首 chunk 超时（30s） | 兜底保护：首 chunk 不到时不再无限等待 |
| 🟡 P1 | parse-stream → stall 改为 reject | 避免"错误伪装成正常回复" |
| 🟢 P2 | llm-bridge → 总超时（5 分钟） | 极限情况兜底 |

## 五、验证方案

### 5.1 模拟"已停止"场景

1. 开始一次长回复（如让模型生成大量代码）
2. 在回复进行中手动刷新 DeepSeek 页面或断开网络
3. 观察 pi 是否能收到错误（而非卡死）
4. 观察 pi 的后续行为（重试、报错等）

### 5.2 模拟首 chunk 超时

1. 临时将 `FIRST_CHUNK_TIMEOUT_MS` 改为 3s
2. 发送一个复杂的 prompt
3. 确认 3s 无首 chunk 时走错误路径

### 5.3 验证 OpenAI 错误协议对齐

1. 使用 curl 或 Python OpenAI SDK 连接本地 `/v1/chat/completions`
2. 触发中断 → 确认 SSE 流中出现 `{"error":{...}}` 事件
3. 确认客户端 SDK 正确抛出 `APIConnectionError`

## 六、总结

通过引入 DOM "已停止"检测作为主要信号，配合 stall watchdog 和首 chunk 超时作为兜底，将 DeepSeek 网页中断的检测从"靠猜测"变为"用 DeepSeek 自己的确定性信号"。同时通过补齐 `onError` 回调和 SSE error 事件写入，让整个错误传播链路对齐 OpenAI 原生 API 的行为，使 pi 无需修改就能正确处理各种中断场景。
